# 패치 방향

## 기본 전략

작은 실험 패치부터 시작한다.

1. 원본 파일을 직접 손으로 고치지 않는다.
2. `main.bundle.js`와 필요한 자산을 백업한다.
3. 치환표 기반 패처를 만든다.
4. 정확히 일치하는 문자열만 바꾼다.
5. 패치 후 게임 실행과 화면 확인으로 sanity check를 한다.

## 번역할 것

- 긴 설명문
- 도움말 문장
- 툴팁
- 효과 설명
- 스탯 표시 라벨
- 터미널 출력 중 플레이어 안내 문장

예시:

- `Effects:` -> `효과:`
- `hacking skill` -> `해킹 스킬`
- `hacking exp` -> `해킹 경험치`
- `Required hacking skill for hack() and backdoor:` -> `hack() 및 backdoor 필요 해킹 스킬:`

## 번역하지 않을 것

아래는 호환성 문제를 만들 수 있으므로 유지한다.

- 서버 이름: `n00dles`, `foodnstuff`, `home`
- faction 이름: `CyberSec`, `NiteSec`, `BitRunners`
- augmentation 이름: `CRTX42-AA Gene Modification`, `NeuroFlux Governor`
- API 함수명: `hack()`, `grow()`, `weaken()`, `scp()`, `backdoor()`
- 실행 명령어: `run`, `analyze`, `connect`, `home`
- 파일명/프로그램명: `NUKE.exe`, `BruteSSH.exe`
- enum/string ID로 쓰이는 값

## 폰트 적용 방향

우선순위:

1. 네오둥근모를 앱 자산으로 복사한다.
2. CSS에 `@font-face`를 추가한다.
3. 기본 폰트 패밀리를 `NeoDunggeunmo, JetBrainsMono, "Courier New", monospace` 형태로 바꾼다.
4. 코드 에디터 입력/터미널/일반 UI에서 한글 폭과 줄높이를 확인한다.

주의: 폰트 파일명을 바꾸기보다는 `assets/fonts/neodgm.ttf`로 관리하고, 패처가 게임 dist 자산에 복사하도록 한다.

## 패치 방식 후보

### A. 직접 번들 치환

가장 빠른 실험 방식이다. `main.bundle.js` 안의 정확한 영어 문자열을 한국어로 바꾼다.

장점:

- 소스 빌드가 필요 없다.
- 작은 범위 실험이 빠르다.

단점:

- Steam 업데이트 때 덮어씌워질 수 있다.
- 번들 구조가 바뀌면 치환 실패 가능성이 있다.

### B. 소스 포크 빌드

공식 저장소를 받아 소스 수준에서 수정하고 Electron 앱을 다시 빌드한다.

장점:

- 구조적으로 가장 깨끗하다.
- TypeScript 기준으로 작업 가능하다.

단점:

- Node 24 이상과 빌드 환경이 필요하다.
- 초기 비용이 크다.

### C. DOM 후처리 번역

런타임에서 화면 텍스트를 찾아 바꾼다.

판단: 실험 우선순위 낮음. React와 충돌할 수 있고, 공식 이슈에도 번역 확장으로 인한 DOM 오류 사례가 있다.

## 채택 방향

초기에는 A 방식을 채택한다. 단, 모든 변경은 패처와 치환표로 남겨 Steam 업데이트 후 재적용 가능하게 만든다. 범위가 커지면 B 방식으로 전환한다.

## 첫 패치 후 방향 수정

`Hacknet Nodes` 소개문 3개 치환이 성공했으므로 초기 전략은 유효하다. 다만 앞으로는 수동 치환을 반복하지 않고, 반드시 패처를 통해 적용한다.

수정된 우선순위:

1. 수동 치환 금지, JSON 치환표 기반 패처 작성
2. 폰트 적용은 번역 패치와 분리해서 실험
3. 화면 단위로 번역 범위 확장
4. 실패한 패치는 즉시 롤백하고 원인을 기록

첫 성공이 의미하는 것:

- 번들 직접 치환은 초기 실험 방식으로 충분하다.
- 한국어 문장 자체는 React 렌더링을 깨지 않는다.
- 고유명사 유지가 문맥과 안정성 양쪽에서 좋다.
- 앞으로의 위험은 번역 자체보다 자동화/롤백/업데이트 대응 쪽에 있다.
